04a — IconographyFoundation

One line-icon family. Quiet, 1.5-weight, never decorative.

Folio standardises on Lucide — a single open-source set so every glyph shares the same grid, stroke and corner language. Icons clarify a label or name an icon-only control; they never carry meaning on their own.

Library
Lucide. The single, standard icon source.
Grid
24×24 viewBox. Drawn on a 2px stroke.
Render at
14 · 16 · 20px. 24px only for empty states.
Colour
currentColor — inherits text colour. No fills.
The standard: Lucide

Lucide is Folio's icon language — not a suggestion. Its drawing carries the same restraint as the rest of the system: an even monoline, gently rounded corners and generous interior space, so a glyph reads as a quiet hairline rather than a mark that wants attention. That's exactly the register the UI keeps — type and spacing lead, ornament recedes — so icons sit beside a label without ever competing with it. It also supersedes the bespoke inline SVGs that were drawn by hand in Nav, Buttons, Overflow menu and Tooltip; those now use the named Lucide equivalents below.

Don't
No second icon library, no filled/duotone variants, no brand or app logos sitting in the icon slot. Don't shrink an icon below 14px or colour it for decoration — colour follows the text it sits beside. An icon never appears without an adjacent label or an aria-label.

12 — Overflow menuPrimitive

Tucks away lifecycle and destructive actions. It never edits.

The overflow menu is for actions that aren't first-class enough to deserve a button, but are too consequential to be hidden in a hover affordance: mark as sold, archive, delete, export. Inline edits live on the field itself (see 13a) — they never appear here. The trigger is a single ghost ⋯ button, anchored to the detail head or a card header. One menu per record; never nested.

Normal items follow the system's universal hover token (also used by ghost buttons, table rows, the entity filter). Destructive items get a 8%-negative fill so their tone survives the hover — without it the red text washes out against the gray fill and the row stops reading as "danger" at the exact moment it should.

When to reach for it
Use the overflow menu
  • State transitions on a record — mark as sold, archive, reopen.
  • Destructive actions on the focused entity — delete property, remove loan link.
  • Rare meta actions — export, view audit history, duplicate.
  • Anywhere you'd otherwise add a fourth header button.
Don't use it
  • Editing a field value — that's inline edit (see 13a).
  • Switching tabs, navigating, opening a related record — use a link.
  • Toggling a boolean the user controls often — use a switch in place.
  • The primary action of the screen — that's a button in the header.

Quiet by default. Reveals only what a glance hides.

Folio favours visible labels and complete sentences over hover-only metadata, so tooltips are rare. They earn their place in two cases: naming an icon-only control, and explaining why a disabled action can't be used.

Use for

Don't use for
Primary information. Anything required to complete a task.
Variants
Inline (single line) · Block (multi-line, max 220px).
Trigger

Markup
Apply data-tooltip="…" to any element. Add .tooltip--block when the message needs to wrap (e.g. the disabled-Delete explanation above). For elements that themselves swallow pointer events (disabled buttons), wrap them in a .tooltip-host span so hover still resolves on the wrapper.

13 — Interaction patternsPattern

How Folio behaves, not just how it looks.

Decisions on editing and confirming that should be the same everywhere in the app. One edit pattern, two confirmation tiers — chosen for a calm, deliberate, single-user product. Each entry below is a decision, not a suggestion.

Rule of one
One edit pattern across the app. Don't mix inline-edit with edit-modes with drawers.
Stakes > speed
If an action takes input, cascades, or can't be reversed cheaply — make the user pause.
Reconsider when
Folio adds bulk operations or a high-velocity triage flow. Until then, YAGNI.
13a · Editing

Inline edit-in-place. One pattern, applied everywhere.

Every editable value reveals an affordance on hover and becomes an input on click. No edit modes, no per-record drawers, no "Save / Cancel" footers.
Core entities (properties, loans, entities) don't change often. When they do, the user is making a single, considered tweak — not filling a form. Inline edit keeps the record visible during the edit and removes a whole layer of UI chrome. Relationship fields (entity, agent) use inline pickers; lifecycle actions (sold, archive, delete) live in the overflow menu and use confirmations — they are not edits.

Use inline edit
  • Single field on a known record (nickname, weekly rent, lease date).
  • Relationship pick (entity, agent) — opens an inline picker, not a modal.
  • Anywhere the user reads the value and the edit; both should stay visible.
Do not use inline edit
  • Creating a record (e.g. adding a property) — that's a dedicated flow.
  • State transitions (mark as sold, archive, delete) — use confirmations.
  • Multi-field forms with cross-validation — rare, but reach for a modal.
Affordance
Hover reveals a subtle surface-sunken background and a small pencil glyph. Editable fields never need a "Click to edit" label — the affordance is the contract. If a field can't display its affordance clearly, the field is not editable in place.
13b · Confirming

Two tiers. Inline for contextual, modal for explicit.

If an action takes input → modal. If it's a single contextual yes/no → inline confirm. If it's irreversible or cascades to other records → always modal.
Confirmations are the moment the app slows the user down. We tier them so the friction matches the stakes: contextual deletes and dismissals stay in place, while state transitions and destructive actions become an explicit, focused decision. Folio does not use toast-with-Undo — see the note below.
